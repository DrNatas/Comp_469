Section 5.2: Formal Game Formulation

A rigorous competitive architecture is defined mathematically as a formal system containing:

States & Turns

$S$: The total set of valid states, beginning at initial root $S_0$.

$\text{TO-MOVE}(s)$: Identifies which specific player acts in state $s$.

Actions & Transitions

$\text{ACTIONS}(s)$: Returns the complete set of all legal moves available in state $s$.

$\text{RESULT}(s, a)$: The transition model mapping actions to states.

Terminals & Payoffs

$\text{IS-TERMINAL}(s)$: A boolean function checking if the game has ended.

$\text{UTILITY}(s, p)$: The precise numeric payoff for player $p$ at terminal node $s$.

Speaker Notes: This formal layout builds on the 5-tuple from Chapter 3, but introduces two crucial updates for adversarial tracking: `TO-MOVE` keeps track of whose turn it is, and `UTILITY` replaces simple path costs with multi-agent payoff evaluations at the terminal leaves.
Slide 6 / 40

The Zero-Sum Property Axiom

In a zero-sum game, the total total utility allocated to all competing players remains constant across all terminal states.

Utility(Max) + Utility(Min) = 0

Any advantage secured by one player represents an equal, direct loss for the opponent, creating an environment of pure, uncompromised competition.

Game Tree Conventions

  • Nodes: Represent exact environmental configurations.
  • Edges: Represent valid player transitions.
  • Max Nodes ($\mathbf{\triangle}$): Transitions controlled by our maximizing agent.
  • Min Nodes ($\mathbf{\nabla}$): Transitions controlled by the minimizing competitor.
Speaker Notes: Zero-sum properties mean there is zero room for cooperation. Max and Min are locked in absolute opposition. When graphing these trees, we use upward-pointing triangles for Max layers and downward-pointing triangles for Min layers to clarify selection directions.
Slide 7 / 40

The Minimax Value Equation

The optimal strategy for a zero-sum game is determined by recursively computing the **Minimax Value** for every state in the tree:

MINIMAX(s) =
    UTILITY(s)                           if IS-TERMINAL(s)
    maxa ∈ ACTIONS(s) MINIMAX(RESULT(s, a))   if TO-MOVE(s) == MAX
    mina ∈ ACTIONS(s) MINIMAX(RESULT(s, a))   if TO-MOVE(s) == MIN

This recursive backing-up of values assumes that both players play with perfect rationality from any state onward.

Speaker Notes: This piecewise definition forms the mathematical core of adversarial search. Value propagation flows from the bottom up. We can't know the value of the root node without first computing the values of the terminal states and passing them back up through the decision tree layers.

Worked Example: Minimax Tracing

Tree Architecture

  • Root Node $A$ (Max turn).
  • Children of $A$: Min Nodes $B$ and $C$.
  • Children of $B$: Leaves $D(3)$ and $E(12)$.
  • Children of $C$: Leaves $F(8)$ and $G(2)$.

Value Backpropagation Trace

  1. Evaluate Min Node $B$: $\min(3, 12) = \mathbf{3}$.
  2. Evaluate Min Node $C$: $\min(8, 2) = \mathbf{2}$.
  3. Evaluate Max Root $A$: $\max(B, C) = \max(3, 2) = \mathbf{3}$.

Decision: Max selects the action leading to $B$. Looking for the 12 at node $E$ is a trap; a rational Min player will force the game to 3.

Speaker Notes: Look at this choice. A common mistake is for an agent to chase the high score of 12 at node $E$. But because it is Min's turn at node $B$, Min will always pick the move that minimizes the score. Minimax correctly anticipates this choice and routes the game through $B$, securing a guaranteed value of 3 instead of risking a drop to 2 at node $G$.

Standard Minimax Implementation

def MINIMAX-DECISION(state):
    return max(problem.ACTIONS(state), key=lambda a: MIN-VALUE(problem.RESULT(state, a)))

def MAX-VALUE(state):
    if problem.IS-TERMINAL(state): return problem.UTILITY(state)
    v = -float('inf')
    for a in problem.ACTIONS(state): v = max(v, MIN-VALUE(problem.RESULT(state, a)))
    return v

def MIN-VALUE(state):
    if problem.IS-TERMINAL(state): return problem.UTILITY(state)
    v = float('inf')
    for a in problem.ACTIONS(state): v = min(v, MAX-VALUE(problem.RESULT(state, a)))
    return v
Speaker Notes: This implementation uses mutually recursive functions: `MAX-VALUE` and `MIN-VALUE`. It performs a full depth-first traversal of the game tree. Notice that it visits every single node down to the terminal leaves, which leads us directly to our next problem: resource consumption.

Asymptotic Scaling Limits of Minimax

Complexity Equations

  • Time Complexity: $O(b^m)$
  • Space Complexity: $O(bm)$

Where $b$ represents the branching factor (legal moves) and $m$ is the maximum depth of the tree.

The Real-World Scale

Chess Bounds: $b \approx 35, m \approx 80 \rightarrow 35^{80}$ leaf nodes.

The universe contains roughly $10^{80}$ atoms. Running an unoptimized Minimax search for chess is physically impossible due to this exponential explosion of states.

Speaker Notes: Raw Minimax is completely impractical for complex real-world games. If your tree search requires more states than there are atoms in the universe, it will never finish running. We need a way to prune the tree without losing our optimality guarantees.

Section 5.3: Alpha-Beta Pruning

Pruning Unproductive Subtrees Optimally

Eliminating entire branches of the search tree when we can prove a rational opponent will never allow them to be reached.

Speaker Notes: We are now entering Section 5.3: Alpha-Beta Pruning. This optimization technique lets us skip expanding massive subtrees without changing the final Minimax decision. It is an exact method, meaning it provides the same optimal move choice as a full tree search, but much faster.
Slide 12 / 40

Anatomy of Alpha and Beta Bounds

We maintain two tracking metrics dynamically during our depth-first traversal of the game tree:

Alpha (α) Parameters

  • Tracks the **highest value** found so far along any choice path for **Max**.
  • Initialized globally to $-\infty$.
  • Max updates $\alpha$ whenever a child node offers a higher guaranteed payoff.

Beta (β) Parameters

  • Tracks the **lowest value** found so far along any choice path for **Min**.
  • Initialized globally to $+\infty$.
  • Min updates $\beta$ whenever a child node reveals a lower alternative payoff.
Speaker Notes: Alpha is Max's best alternative, and Beta is Min's best alternative. As we dive deeper into the tree, these bounds tighten. If Alpha ever becomes greater than or equal to Beta, it means the current path is worse than a previously discovered option for one of the players, allowing us to safely prune the remaining branches.
Slide 13 / 40

The Alpha-Beta Pruning Axiom

The standard condition for pruning a branch is absolute and immediate:

Prune immediately if: α ≥ β

Max Layer Pruning

If a Max node discovers a child value greater than or equal to the current $\beta$ bound, stop expanding its remaining siblings. Min further up the tree would never allow this path to occur.

Min Layer Pruning

If a Min node discovers a child value less than or equal to the current $\alpha$ bound, stop expanding its remaining siblings. Max further up the tree would choose a better alternative path elsewhere.

Speaker Notes: Understand this deeply: we prune because we assume rational play. If Max can force a score of 5 on a different branch, and the current Min node reveals a child value of 3, Max will never pick the path leading to this Min node. There's no reason to waste compute power looking at the rest of this Min node's options.

Alpha-Beta Pruning Code Engine

def ALPHA-BETA-SEARCH(state):
    v = MAX-VALUE(state, -float('inf'), float('inf'))
    return best_action_matching_v

def MAX-VALUE(state, alpha, beta):
    if problem.IS-TERMINAL(state): return problem.UTILITY(state)
    v = -float('inf')
    for a in problem.ACTIONS(state):
        v = max(v, MIN-VALUE(problem.RESULT(state, a), alpha, beta))
        if v >= beta: return v  # Prune remaining sibling branches
        alpha = max(alpha, v)
    return v

def MIN-VALUE(state, alpha, beta):
    if problem.IS-TERMINAL(state): return problem.UTILITY(state)
    v = float('inf')
    for a in problem.ACTIONS(state):
        v = min(v, MAX-VALUE(problem.RESULT(state, a), alpha, beta))
        if v <= alpha: return v  # Prune remaining sibling branches
        beta = min(beta, v)
    return v
Speaker Notes: This code extends our standard Minimax recursive loop by passing and updating `alpha` and `beta` downward. The early return statements (`return v`) break out of the expansion loop entirely, skipping remaining siblings and saving massive amounts of processing time.

Worked Numerical Trace: Alpha-Beta

Tree Node Architecture

  • Max Root $A$. Children: Min Nodes $B$ and $C$.
  • Children of $B$: Leaves $D(4)$ and $E(5)$.
  • Children of $C$: Leaves $F(2)$ and $G(9)$.

Step-by-Step Traversal Trace

  1. Evaluate $B$: Check $D(4) \rightarrow \beta=4$. Check $E(5) \rightarrow \beta$ stays 4. $B$ returns 4.
  2. Root $A$ updates its $\alpha = \max(-\infty, 4) = \mathbf{4}$.
  3. Evaluate $C$ with parameters $\alpha=4, \beta=+\infty$.
  4. Check leaf $F(2)$. Min node $C$ updates its $\beta = \min(+\infty, 2) = \mathbf{2}$.
  5. Check Pruning Axiom: Is $\alpha \ge \beta$? $4 \ge 2 \rightarrow$ **True**.
  6. Prune Node $G$! Return 2 from $C$ immediately without ever expanding $G$.
Speaker Notes: Let's follow this trace carefully. Once node $C$ sees the 2 at leaf $F$, it knows its final value will be 2 or lower. Since Max already has a guaranteed option of 4 at node $B$, Max will never choose node $C$. We can safely prune node $G$ without ever looking at its value of 9.

The Crucial Role of Move Ordering

Perfect Move Ordering

If choices are sorted so that Max examines the best moves first and Min examines the worst moves first, Alpha-Beta efficiency peaks.

Optimal Time Complexity = O(bm/2)

This effectively doubles our search depth, allowing an agent to look twice as far ahead using the same compute budget.

Pessimistic Ordering

If moves are sorted in the worst possible order, Alpha-Beta finds zero pruning opportunities.

Worst Case Complexity = O(bm)

The algorithm is forced to evaluate every single leaf node, collapsing back to the performance of raw Minimax.

Speaker Notes: Move ordering is what makes Alpha-Beta pruning work in practice. Under perfect ordering, our time complexity drops to $O(b^{m/2})$. In other words, you can search twice as deep in the same amount of time. This shows why sorting moves using basic heuristics before pushing them to the frontier queue is so critical.

Section 5.4: Imperfect Real-Time Decisions

Heuristic Truncation Under Strict Time Bounds

Modifying search frameworks to enforce real-time processing constraints by replacing terminal evaluations with fixed depth cutoffs.

Speaker Notes: Even with Alpha-Beta pruning, we cannot search all the way to the end of a game like chess within a reasonable time limit. Section 5.4 addresses this by introducing heuristic evaluation functions that estimate state values at a fixed depth cutoff, turning our ideal search into a real-time decision engine.
Slide 18 / 40

The Cutoff Search Modification

To operate safely within real-time time boxes, we alter our search mechanics using two core modifications:

1. The CUTOFF-TEST Strategy

Replaces the standard IS-TERMINAL evaluation. It returns True if the game is over OR if the current search depth hits a predefined limit $d$.

2. The EVAL Heuristic Function

Replaces the true UTILITY function at cutoff points, returning an estimate of the expected utility from that state onward.

Altered Condition: if CUTOFF-TEST(state, depth): return EVAL(state)
Speaker Notes: This change turns our exhaustive search into an approximation. Instead of evaluating true wins or losses at terminal leaves, we stop searching at a fixed depth limit and use `EVAL` to guess how good the position is for our agent.
Slide 19 / 40

Designing Linear Evaluation Functions

A standard evaluation function computes a weighted linear combination of distinct state features:

EVAL(s) = w1f1(s) + w2f2(s) + ... + wnfn(s)

Chess Evaluation Example

  • Features ($f_i$): Material counts (Pawn=1, Knight=3, Rook=5, Queen=9).
  • Weights ($w_i$): Strategic values assigned to each piece based on position or developmental stage.
  • Positive scores favor Max; negative scores favor Min.
Speaker Notes: Linear evaluation functions are incredibly efficient to compute. In chess, for example, features represent piece counts, and weights reflect their relative value. A positive total score means Max is ahead, while a negative score means Min holds the advantage.
Slide 20 / 40

The Horizon Effect & Quiescence

The Horizon Effect

Occurs when a severe, unavoidable threat is pushed just past the search depth limit (the horizon). The agent remains oblivious to the danger because it cuts off search right before the trap snaps shut.

Quiescence Search Solution

Prevents cutting off search during volatile, unstable game states (like a piece capture in progress).

Rule: Only apply `EVAL` to *quiescent* (stable) states. If a state is volatile, extend the search depth automatically until positions stabilize.

Speaker Notes: The horizon effect is a classic pitfall in adversarial search. If an agent's search depth is capped at 4, it might make a move that delays a major loss until turn 5, thinking it safe. We mitigate this with a quiescence search, extending the search deeper along volatile paths to ensure our evaluation estimates are stable and accurate.

Section 5.5: Stochastic (Chance) Games

Modeling Environments with Random Outcomes

Integrating random events—such as rolling dice or shuffling cards—directly into our game tree search using Expectiminimax.

Speaker Notes: We now move to Section 5.5: Stochastic Games. Games like Backgammon or Monopoly introduce elements of chance that cannot be predicted deterministically. To handle this randomness, we introduce a new type of node to our game tree: the Chance Node.
Slide 22 / 40

The Expectiminimax Tree Structure

Stochastic game trees alternate between Max nodes, Min nodes, and **Chance nodes** (represented as circles):

The Probabilistic Calculation

Instead of choosing the max or min value among its children, a Chance node computes a weighted average based on the probabilities of each random outcome occurring:

EXPECTIMINIMAX(Chance Node) = Σr P(r) × EXPECTIMINIMAX(RESULT(s, r))
Speaker Notes: Notice how the structural logic changes. A chance node doesn't pick a favorite child. Instead, it calculates the mathematical expectation across all possible random outcomes, multiplying the probability of each event by its resulting node value.
Slide 23 / 40

Worked Example: Expectiminimax

Tree Architecture Setup

  • Max Root Node $A$.
  • Action choices lead to Chance Nodes $B$ and $C$.
  • Chance Node $B$ paths: Outcome 1 [Prob 0.5] leads to value 10. Outcome 2 [Prob 0.5] leads to value 2.
  • Chance Node $C$ paths: Outcome 1 [Prob 0.9] leads to value 3. Outcome 2 [Prob 0.1] leads to value 40.

Expectation Computations

  1. Evaluate Chance Node $B$:
    $(0.5 \times 10) + (0.5 \times 2) = 5 + 1 = \mathbf{5}$.
  2. Evaluate Chance Node $C$:
    $(0.9 \times 3) + (0.1 \times 40) = 2.7 + 4 = \mathbf{6.7}$.
  3. Evaluate Max Root $A$:
    $\max(B, C) = \max(5, 6.7) = \mathbf{6.7}$.

Max chooses the path to node $C$ to maximize expected value.

Speaker Notes: Walk through this math step by step with the class. Chance node $B$ yields an expected value of 5, while node $C$ yields 6.7. Even though node $B$ offers a potential payout of 10, the high probability of landing a low score at node $C$ is offset by its outlier payout of 40, making $C$ the mathematically superior strategic choice.

Complexity and Pruning Bounds of Chance

Complexity Scaling Expansion

Complexity = O(bm × nm)

Where $n$ represents the total number of distinct random outcomes possible at each chance node step.

The Pruning Barrier

  • Pruning branches under chance conditions is incredibly difficult.
  • Because chance nodes calculate averages rather than simple minimums or maximums, a single unexamined leaf node with a massive outlier value can completely shift the expected value of a branch.
Speaker Notes: Randomness causes our search trees to explode in size, with a time complexity of $O(b^m \cdot n^m)$. Alpha-Beta pruning becomes much less effective here because an unvisited leaf node hidden under a chance branch could have a massive value that completely changes the calculated average.
Slide 25 / 40

Sensitivity to Heuristic Scale Transformations

Minimax Scale Invariance

Standard Minimax decisions are completely invariant under any monotonic, order-preserving transformations of the evaluation scale (like multiplying all scores by 10 or adding 5).

Expectiminimax Scale Sensitivity

Stochastic searches are highly sensitive to linear scaling transformations. Modifying the evaluation scale changes the calculated averages, which can completely alter the agent's preferred move choices.

Rule: When designing heuristics for stochastic games, your evaluation values must map strictly and linearly to the true probabilities of winning.

Speaker Notes: This is a crucial theoretical distinction. In pure Minimax, changing your evaluation values from 1, 2, 3 to 1, 10, 100 doesn't change the final decision because the relative ordering of moves remains identical. In Expectiminimax, however, changing the scale alters the calculated averages, which can completely shift your agent's preferred strategy.

The Foundation of Belief States

Belief State Modeling

  • The agent can no longer isolate its exact location down to a single definitive node point in the tree.
  • Instead, it evaluates a **Belief State**: the set of all possible actual game states that are consistent with the cards, board layouts, and move histories observed so far.

Kriegspiel Chess Example

A variant of chess where you cannot see your opponent's pieces. Your belief state contains thousands of alternative board layouts where the enemy's pieces could legally be positioned given the historical move announcements.

Speaker Notes: Think of a belief state as a cloud of possibilities. When playing poker, your belief state contains every possible combination of cards your opponent could be holding given the cards face up on the table and their betting behavior. Your strategic choices must perform well across this entire cloud of possibilities.

The Trap of "Average Case" Approximations

The Deterministic Trap Blueprint

A common mistake when designing card-game AI is to generate a single "average" board layout by guessing the hidden items, solve that specific layout using standard Minimax, and assume the resulting move is optimal.

This approach fails completely because it ignores the strategic value of **information gathering** actions, bluffs, and defensive play designed to protect hidden data.

Speaker Notes: Do not fall into the average-case trap. If you assume a single average hand for your opponent, your AI will never learn to play defensively against outliers, bluff effectively, or make moves purely to gather information about the opponent's hidden state.
Slide 29 / 40

The Strategy of Information Gathering

Active Epistemic Actions

Moves executed specifically to reveal hidden data and shrink the size of the agent's belief state cloud.

  • Executing a probe bet in poker to test an opponent's strength.
  • Firing at target coordinates in Battleship to find enemy assets.
  • Sacrificing a minor piece in Kriegspiel to force a referee declaration.

Mathematical Value

The strategic value of an information-gathering action is measured by how effectively it reduces the entropy of the agent's belief state, paving the way for safer, more optimal decisions on subsequent turns.

Speaker Notes: Information is a tangible resource in partially observable games. Sometimes, the mathematically optimal move is not the one that scores points directly, but the one that forces your opponent to reveal their hidden cards or positions, significantly simplifying your future planning.

Game Theory Matrix Representation

When hidden state variables prevent us from using explicit tree graphs, we evaluate payoffs using a **Payoff Matrix**:

Pure vs. Mixed Strategy Profiles

Pure Strategies

The agent commits 100% of its choices to a single deterministic action path.

Risk: Highly exploitable in partially observable environments, as an opponent can decode your deterministic pattern and design a direct counter-strategy.

Mixed Strategies

The agent chooses actions probabilistically based on a carefully calculated optimal distribution.

Distribution Profile: [W: 0.60, X: 0.40]

This introduces optimal unpredictability, ensuring a guaranteed baseline expected return regardless of how the opponent responds.

Speaker Notes: Mixed strategies are mandatory in games of imperfect information like Rock-Paper-Scissors or Poker. If your strategy is pure and deterministic, a smart opponent will decode your pattern and exploit it. By playing a mixed strategy based on an optimal probability distribution, you make your play unexploitable.

The Transposition Table Cache

A **Transposition Table** caches previously evaluated state positions within a global hash map to prevent re-computing identical positions reached via alternative move sequences.

Identical positions reached via different paths are called transpositions (e.g., reaching the same board state via move order A then B versus move order B then A).

Cached Data Record

  • The exact hash key of the game state configuration.
  • The backed-up Minimax value or bounding parameters ($\alpha, \beta$).
  • The maximum depth evaluated along that branch.
Speaker Notes: Transposition tables are essentially large-scale memoization tools for game trees. In many games, different sequences of moves lead to the exact same board position. By caching these results in a hash table, we can reuse the previously calculated values instantly instead of re-searching the entire subtree.

The Mechanics of Forward Pruning

Standard Alpha-Beta Pruning

Only eliminates branches that are *provably* worse than existing options, ensuring zero risk of missing the optimal move.

Forward Pruning Strategy

Intentionally discards branches that look highly unlikely to be good based on early heuristics, sacrificing absolute guarantees to save massive compute time.

Risk: Risks missing a brilliant, counter-intuitive move that appeared weak during initial shallow evaluations.

Speaker Notes: Forward pruning is where we trade absolute mathematical certainty for raw speed and depth. Instead of waiting for a rigorous proof to prune, we use heuristics to discard unpromising lines early on. This lets our search run significantly deeper, but introduces the risk of overlooking a brilliant, hidden masterstroke move.
Slide 35 / 40

The PROBCUT Search Algorithm

The **PROBCUT** (Probabilistic Cut) algorithm uses historical game data and statistics to optimize forward pruning decisions.

It performs a quick, shallow search to estimate a node's value ($v$), then uses past statistical data to calculate the probability that a full-depth search would yield a score outside the current $[\alpha, \beta]$ window.

Historical Performance

When applied to the Othello program LOGISTELLO, adding PROBCUT enabled the engine to defeat its standard, non-probabilistic counterpart 64% of the time, even when the standard version was granted twice as much processing time.

Speaker Notes: PROBCUT brings statistics to adversarial search. Instead of spending valuable compute time verifying an option down to depth 10, it runs a quick search to depth 2, looks up past game data to see how likely that score is to be useful, and prunes the entire branch immediately if the odds are low.

Late Move Reduction (LMR)

Core Underlying Assumption

LMR assumes that if your move-sorting heuristic is highly effective, moves that appear late in the sorted list are unlikely to be optimal.

The Search Depth Reduction

  • Instead of pruning late options completely, LMR searches them to a **reduced depth**, saving massive amounts of time.
  • If a reduced-depth search unexpectedly returns a high score above $\alpha$, the engine catches the anomaly, resets parameters, and re-runs a full-depth search along that branch.
Speaker Notes: Late Move Reduction is a highly effective compromise. We don't discard late, unpromising moves completely; we just search them shallowly to save time. If a shallow search reveals a surprise opportunity, our engine catches it, scales up the search depth, and evaluates the path thoroughly.
Slide 37 / 40

Monte Carlo Tree Search (MCTS)

When a game's massive branching factor makes traditional evaluation functions impossible, we turn to MCTS:

The 4-Step MCTS Core Loop

  1. Selection: Traverse down known nodes using selection rules like UCB1.
  2. Expansion: Create a new child leaf node to extend the tree.
  3. Simulation (Rollout): Play out the game completely to the finish line using random or semi-random choices.
  4. Backpropagation: Use the rollout's win/loss outcome to update statistical records along the path.

The AlphaGo Breakthrough

MCTS forms the core architecture behind AlphaGo, bypassing the need for handcrafted chess-like evaluation functions by using deep neural networks to guide rollouts and selection paths across Go's massive state space.

Speaker Notes: Monte Carlo Tree Search revolutionized AI for games like Go, where traditional evaluation functions fail because even a single stone placement changes the entire board dynamics. Instead of guessing a state's value, MCTS plays out thousands of rapid, random games to the finish line, using the actual win-loss ratios to determine the best move.
